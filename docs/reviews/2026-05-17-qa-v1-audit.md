### F-2 ‚Äî `/me` persona picker switch + apply works correctly (positive)

**P2 ‚Äî confirming healthy.** Default persona "Marketing Director ¬∑ Robotics & AI" ‚Üí after Apply, the page updated to "Engineering Manager ¬∑ Backend Infrastructure" with new owner role, About-me, and a fresh system_prompt (`make_pdf`, `decompose_task`, `ambiguity_probe` are persona-specific). ### F-3 ‚Äî `/` chat send button is click-blocked by Next.js dev overlay

**P1 (dev-only) ‚Äî likely production-fine but blocks automation.** The `.chat-send` button is visible, enabled, and stable, but `<nextjs-portal>` from `data-nextjs-dev-overlay="true"` intercepts pointer events for the full viewport. Manual testing with `Enter` keypress should still send. Playwright tests need to either `.press('Enter')` or `force: true` click. This will not affect a production build.

**Recommended fix:** none required; document as a dev-mode quirk in the E2E test files (use `Enter` to submit).

### F-4 ‚Äî `/me` "Identity ‚Üí Owner name" empty after fresh load

**P2.** The `Owner name` field shows placeholder text ("What should the desk AI call you?") rather than a saved value. The MEMORY.md context names the user "chen.zhang6@gmail.com" but the desk has no resolved identity. Acceptance for Flow 1 ("Desk AI greets within 1.5s using snapshot context") is satisfied in the chat surface, but the greeting will be generic. Fine for V1; flag if persona-onboarding is on the roadmap.

### F-5 ‚Äî `/inbound` and `/deliverables` correctly render empty state (positive)

**P2 ‚Äî confirming healthy.** Both pages render the heading + filter chips + empty-state copy. ### F-6 ‚Äî `/members` is empty per fixture; `+ Hire` CTA present on desktop

**P2.** Detected 0 actual staff cards (the `[class*="staff"]` matches in the raw findings were chip elements, not cards). The "No all members yet" empty-state copy is correct. The `+ Hire` button IS present on the filter strip, top-right. **Mobile is broken ‚Äî see M-1 below.**

### F-7 ‚Äî One uncaught Next.js error pill ("1 Issue") on mobile members route

**P1.** A red "1 Issue" pill (Next.js dev runtime error indicator) appeared bottom-right on `/members` at the 390√ó844 viewport. Did NOT appear on desktop. Likely caused by the CliTerminalLauncher / xterm trying to mount in a narrow viewport. The "1 issue" indicator only appears in dev; needs production-build verification before declaring P0.

**Recommended action:** open the dev overlay, read the error, file the bug (or repro in production build to discard the warning).

### F-8 ‚Äî `DEEPSEEK_API_KEY not configured` on Hire-dialog Generate (and any LLM round-trip without a key)

**P1.** When clicking `+ Hire` on `/members` ‚Üí typing a sketch ‚Üí `ü™Ñ Generate profile`, the dialog renders the error "DEEPSEEK_API_KEY not configured" under the textarea. The Hire flow is unreachable without an API key, but: the `/skills`, `/templates`, `/references` describe-mode generations succeeded on the same audit run, which means **those routes have a different key-resolution path** (possibly an in-process fallback) than `/api/v1/admin/polish`. Asymmetric key handling is a bug. Either both paths should require the key, or neither.

**Repro:** on a desk with no `DEEPSEEK_API_KEY` env var: open `/members` ‚Üí `+ Hire` ‚Üí fill sketch ‚Üí Generate profile.

**Recommended fix:** unify the LLM key resolution in `apps/web/lib/deepseek-json.ts` so all callers fail with the same error wording and the same surface (modal-inline vs toast).

### F-9 ‚Äî `HireDialog` is missing `role="dialog"` / `aria-modal` attributes

**P2 ‚Äî a11y gap.** Unlike the BugReportButton modal and the catalog `+ New` modals (which both set `role="dialog" aria-modal="true" aria-label="‚Ä¶"`), `HireDialog` uses raw `<div className="bug-modal">`. Screen readers will not announce it as a dialog. This blocked our initial selector strategy (`[role="dialog"]`); had to fall back to `.bug-modal`. Add the three aria attrs to match the other modals.

## Aesthetic findings (what looks unfinished)

### A-1 ‚Äî Describe modal: textarea + preview card render with near-black backgrounds, hiding content

**P0 ‚Äî visual regression.** On all three catalog surfaces (`/skills`, `/templates`, `/references`) the `+ New` modal renders the Describe-tab textarea and the post-generate preview card with very dark backgrounds (#1a1a1a-ish). The text inside (LLM-generated descriptor: name, kind, tags, body, URL for references) is dark grey on near-black ‚Äî barely legible. Compare with the screenshot at `tmp/qa-20260517-v1/skills-modal-generated.png`.

This is likely a CSS variable inheritance issue ‚Äî the modal `<dialog>` is inheriting a `bg` variable that was meant only for code blocks, or the textarea / preview wrapper is missing a `bg-surface` reset.

**Repro:** Open any of the three pages ‚Üí `+ New` ‚Üí Describe tab ‚Üí type anything ‚Üí Generate. The preview that renders is unreadable.

**Effort estimate:** small ‚Äî likely a 1‚Äì3 line CSS fix in the page-level stylesheet.

### A-2 ‚Äî `/skills`, `/templates`, `/references`, `/inbound`, `/deliverables`, `/members`: top-right "√ó close panel" button overlaps the count text

**P1.** The 14-px √ó icon (collapse panel ‚Üí back to chat) sits at the absolute right edge of the page-strip and visually overlaps the count text (e.g. "1 yours ¬∑ 30 examples", "0 items", "0 total") on every panel page. Confirmed on inbound, deliverables, members, skills, references, templates. The text appears to be partially cut by the button hit area.

**Recommended fix:** add right padding to the page-strip count or push the close button further into the corner.

### A-3 ‚Äî `/me` desk: My desk label clipped

**P2.** Right of the persona pill, the label "desk: My desk" overlaps with the same `√ó` close panel button as A-2.

### A-4 ‚Äî Empty-states are correct but minimal ‚Äî no "what to do next" CTA

**P2.** Inbound, deliverables, members, today (untested) all show "No X yet" + minimal copy. The deliverables empty-state has a one-line hint ("Dispatch a job to your staff to produce one") ‚Äî good. Inbound has no such hint. Members has no hint. The chat surface's suggestion chips ("show inbound missions", "list recent deliverables") effectively fill this gap, but only when chat is visible. Worth a content audit pass.

## Mobile findings (390√ó844)

### M-1 ‚Äî `+ Hire` button COMPLETELY MISSING on mobile `/members`

**P0 ‚Äî functional regression.** The horizontal filter-chip row (All / Peer / Virtual / Linked / CLI) overflows the viewport width and the `+ Hire` button is clipped off the right edge. The only visible right-side action is the `√ó` close-panel button. **Mobile users cannot hire any virtual staff** ‚Äî the user explicitly identified this is the daily-flow gap they want closed.

**Repro:** open `/members` on a phone (or 390√ó844 viewport).

**Recommended fix:** make the filter-chip row horizontally scrollable, or move `+ Hire` to a sticky FAB on mobile, or wrap chips on mobile breakpoints.

### M-2 ‚Äî Mobile nav: rightmost tab "References" is clipped

**P1.** The top-nav tab strip overflows at 390px. "References" is partially cut. No horizontal scroll affordance is visible. The bug button (top-right, normally) is also pushed off-screen on the home page mobile screenshot.

**Recommended fix:** allow nav to horizontally scroll with momentum, OR collapse less-used tabs behind an overflow menu at narrow breakpoints.

### M-3 ‚Äî Mobile split-shell: chat composer is shoved into ~225-px width

**P1.** On `/skills` mobile, the chat surface lives on the left + the page panel on the right, both squeezed into 390-px. The chat-input wraps to 2 lines ("Message your desk AI‚Ä¶ (‚åòM / Ctrl+M for voice)") and the send button takes a sizeable chunk. Functional but visually cramped. **The split-shell pattern doesn't work below ~700px**; the app should likely collapse to chat-only or page-only on narrow viewports.

**Recommended fix:** force chat-only or page-only at narrow breakpoints (tab-style switcher in the nav, not split).

### M-4 ‚Äî Mobile suggestion-chips row scrolls horizontally without affordance

**P2.** On `/members` mobile, the chat panel's "show inbound missions / who is on my team? / list recent deliverables / open today's queue" chips overflow horizontally. No fade-on-edge or scroll indicator. Easy to miss the additional chips.

### M-5 ‚Äî "1 Issue" Next.js error pill on mobile `/members`

**P1.** See F-7 above; calling out separately because mobile is where it manifested.
